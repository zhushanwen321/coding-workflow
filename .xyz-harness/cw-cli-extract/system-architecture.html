核心结论
  • 适配层替换，engine 不动。本轮不是重新设计架构，而是替换最外层适配层（pi registerTool 薄壳 → CLI 入口），engine 三层（dispatch/handler → state-machine + checks → store/gates adapter）原样搬迁。
  • 核心计算 = 状态机转换 + 机器检查 gate 执行（技术流程编排，非复杂业务规则），保持现有三层风格，不引入 DDD 4 层。
  • 变化轴正交验证拆分合理性：适配层变化轴=「调用方式/runtime」，engine 变化轴=「业务流程/校验规则」。
  • Seam 纪律：ActionDeps 是真 seam（store/git/runner 已被 mock 证明可替换）；CLI 协议 Port 当前为「假设 seam」（单实现，MCP 落地再升格）。
  • 新建 cli.ts（~150 LOC）+ protocol.ts（~80 LOC）+ 改写 resolveDbPath（~20 LOC），engine 现有 LOC 不变。

设计立场

核心计算是什么？ —— 状态机转换 + 机器检查 gate 执行。这是技术流程编排，不是复杂业务规则编排。

因此分层保持现有风格（不引入 DDD 4 层）：engine 已是清晰的三层结构（dispatch/handler → state-machine + checks → store/gates adapter）。本轮工作不是重新设计架构，而是替换最外层适配层（pi tool 注册 → CLI 入口），engine 三层原样搬迁。

一句话：适配层替换，engine 不动。

挑战与决策

D-A · 适配层是否抽象为多 runtime port？confirmed · D-不可逆

决策：不抽象。本轮单 CLI 实现。

张力：未来要支持 MCP，是否本轮就抽 RuntimeAdapter interface？理由：一个实现=假设 seam，两个才=真 seam（Seam 纪律）。用户已确认 MCP 留后续 topic。过早抽象会产生只有一个实现的 interface（Port≠interface 反面教材）。MCP 落地时再升格。

D-B · 存储路径策略 confirmed by ask_user · D-002

决策：dbPath = ~/.cw/<encoded-cwd>/_cw.json

env CW_HOME 可覆盖根目录。encodeCwd 逻辑复用。pi 现有数据留在 ~/.pi/agent/cw/ 不迁移，两工具数据隔离。理由：抽离成独立工具就该有独立数据空间；数据隔离干净避免迁移风险。

D-C / D-D · CLI 协议风格与大 JSON 传递 confirmed by ask_user · D-001

决策：子命令风格 + stdin/--xxx-file 双通道

cw create --slug X 子命令风格；大 JSON 字段（planJson/clarifyJson/detailJson）走 stdin pipe（主通道）或 --xxx-json-file path（显式文件通道）。exit code 映射 gate 结果。理由：LLM 写子命令比构造 JSON payload 自然；双通道避命令行长度限制；否决 JSON-RPC（大 JSON 超长度+转义易错）。

D-E · nextAction.skill 在非 pi agent 下的语义 confirmed · D-可逆 · D-005

决策：原样透传 skill 字段，CLI 不额外处理

理由：最小行为不改 engine；pi agent 用得上 skill 映射，非 pi agent 靠 nextAction.action + guidance 文本决策。CW 是状态机+gate 非 agent 教程，职责边界清晰。

D-F · ADR-029 worktree cwd 防护去向 confirmed · D-可逆

决策：CLI adapter 继承 pi 的 ADR-029 D1 防御

检测 process.cwd() 含 .cw-wt/ 时拒绝 fallback，强制显式 --workspace 或 CW_WORKSPACE_ROOT env。张力：这是 pi-workflow-specific 防御，通用 CLI 是否需要？理由：保留。成本极低（~10 行）；pi 仍是首个接入方；任意 agent 都可能在 worktree 里 spawn cw，静默丢弃会让 multi-workspace 用户重踩数据隔离坑。泛化为可配置 worktree-prefix 检测留后续。

## 详细分析

### Issue #1: 如何复用 engine 源码而不引入 pi 运行时依赖 — 方案 A（物理拷贝）

#### ⚠️ 安全影响

**风险**: engine 从 pi 扩展薄壳中剥离后，原 pi 工具框架的进程沙箱/权限边界消失；CLI 作为独立 Node 进程拥有运行用户的完整文件系统权限，engine 内部 API 通过子进程协议暴露给任意调用 agent。
**影响范围**: 所有 CLI 子命令、CwStore 文件读写、GitValidator 子进程调用。
**缓解方案**: CLI 不暴露超出 engine 原有能力的接口；所有参数经 typebox schema 校验；存储路径限定在用户目录（默认 `~/.cw/`），env 覆盖仅限 `CW_HOME`/`CW_WORKSPACE_ROOT`；不将用户输入拼接为 shell 命令。
**残余风险**: 任意调用 agent 仍拥有运行用户的文件系统权限；这是本地 CLI 工具的设计预期，非 CW 可直接修复。

#### ⚠️ 可观测性影响

**风险**: pi 扩展原有的 `renderSummary` / TUI 输出被移除，调试信息形态改变；开发者无法再用 pi 的文本输出快速查看 topic 状态。
**影响范围**: 所有命令输出、人类调试体验。
**缓解方案**: CLI 输出统一为结构化 JSON（stdout）；stderr 承载人类可读错误；`status`/`list` 命令提供人读摘要；保留 `ActionResult.details` 中已有的 gateHistory/mustFix 信息。
**残余风险**: 非 JSON 工具链调试时略显不便；这是协议选择可接受的结果。

#### ✅ 数据 / 性能 / 并发 / 稳定性 / 兼容性

- 数据：源码搬迁不修改持久化格式或数据契约。
- 性能：无运行时路径变化，开销与 pi 扩展一致。
- 并发：无并发相关改动。
- 稳定性：物理拷贝最忠实，行为等价性最强。
- 兼容性：engine 源码原样复用，保持 100% 行为等价。

### Issue #2: 如何构造 CLI 适配层入口 — 方案 A（cli.ts + protocol.ts 两层结构）

#### ⚠️ 安全影响

**风险**: 子进程协议 + 大 JSON 输入，存在参数注入、命令注入、JSON 解析异常/DoS 等风险；若校验层与路由层边界不清，容易绕过校验。
**影响范围**: CLI 参数解析、stdin/`--xxx-file` 读取、JSON 校验、action 路由。
**缓解方案**: 所有参数由 `protocol.ts` 统一经 typebox schema 校验；大 JSON 走 stdin/文件，不拼接 shell 命令；不执行用户输入的任意路径作为命令；JSON 解析失败时 exit ≠ 0 并输出 stderr；两层结构确保路由不绕过校验。
**残余风险**: 超大 JSON 仍可能消耗内存，由 #4 性能项单独缓解。

#### ⚠️ 稳定性影响

**风险**: 子进程模型下，未捕获异常或错误映射不当会导致调用方无法区分「程序错误」与「gate fail」；CLI 进程崩溃或挂起会影响调用方 agent 的可靠性。
**影响范围**: 所有子命令、调用方 agent 的调用逻辑。
**缓解方案**: `cli.ts` 顶层统一 try/catch；区分 gate fail（exit 0 + `JSON.gatePassed=false`）与程序错误（exit ≥1 + stderr 人类可读）；未捕获异常走 stderr 并退出非零；不吞掉 engine 抛出的错误。
**残余风险**: 调用方未读取 stdout 仍可能误用 exit code；通过协议文档与 AC-2.4 约束。

#### ⚠️ 可观测性影响

**风险**: stdout 必须纯 JSON，日志/调试信息无法直接打印到 stdout；stderr 是唯一的错误信息通道，输出格式不一致会导致调用方解析困难。
**影响范围**: 所有命令输出、调用方解析逻辑。
**缓解方案**: stdout 始终为 JSON；stderr 仅输出人类可读错误，不输出机器消费字段；`ActionResult` 包含 gateHistory/mustFix/nextAction 等完整状态信息。
**残余风险**: 调用方需同时解析 stdout 与 stderr，这是协议要求。

#### ✅ 数据 / 性能 / 并发 / 兼容性

- 数据：CLI 层只透传参数，不修改数据契约。
- 性能：两层结构文件小，无额外 I/O。
- 并发：无并发相关改动；CLI 进程单次调用。
- 兼容性：新增 CLI 协议，不破坏 engine 接口。

### Issue #3: 如何替换 pi 专用存储路径与 worktree 约定 — 方案 A（完整路径结构 + 双覆盖）

#### ⚠️ 安全影响

**风险**: `CW_HOME` env 与 `--workspace` flag 允许用户覆盖存储根路径，若不对路径进行校验，可能导致路径穿越或写入非预期目录。
**影响范围**: `CwStore` 初始化、`~/.cw/<encoded-cwd>/_cw.json` 读写路径。
**缓解方案**: 仅接受绝对路径；拒绝包含 `..` 或超出用户主目录的覆盖路径；`encodeCwd` 规则固定，cwd 编码后作为子目录；`.cw-wt/` 检测拒绝 fallback，强制显式指定。
**残余风险**: 运行用户仍可覆盖到自己能访问的任意位置，这是本地 CLI 的权限模型。

#### ⚠️ 数据一致性影响

**事务边界**: 单 topic 的 `_cw.json` 写操作由 `CwStore` 现有的 temp+fsync+rename 保证原子性；路径迁移不涉及跨文件事务。
**并发场景**: 多 CLI 进程同时写同一 `_cw.json` 存在覆盖风险；依赖 Node 文件系统无显式锁，但原 engine 已用原子写降低破坏面。本轮不新增并发控制，沿用现有策略。
**迁移方案**: 新路径 `~/.cw/` 与原 `~/.pi/agent/cw/` 隔离，不迁移旧数据；用户需手动处理旧 topic。
**回滚策略**: 路径参数化后，可通过设置 `CW_HOME` 切换数据目录；但 topic 记录不可回滚到 pi 路径。

#### ✅ 性能 / 并发 / 稳定性 / 兼容性 / 可观测性

- 性能：路径计算为纯函数，开销可忽略。
- 稳定性：原子写机制不变，稳定性与 engine 一致。
- 兼容性：不迁移 pi 数据，新 CLI 与 pi 扩展数据独立，互不破坏。
- 可观测性：路径变更不影响日志/指标形态。

### Issue #4: 大 JSON 参数传递机制 — 方案 A（stdin 优先，文件 fallback，同时存在冲突）

#### ⚠️ 安全影响

**风险**: `--xxx-file` 读取用户指定文件，若路径校验不足，可能读取任意文件（如敏感文件）并回显到 stdout 或触发错误泄露。
**影响范围**: `--plan-json-file`、`--clarify-json-file`、`--detail-json-file` 等文件参数。
**缓解方案**: 仅读取后作为 JSON 解析，不执行文件内容；解析失败时 exit ≠ 0；不将文件路径作为 shell 命令执行；文件大小限制见性能项。
**残余风险**: 调用 agent 可读取运行用户能读的文件，这是本地 CLI 权限模型。

#### ⚠️ 性能影响

**预期负载**: CLI 单次调用，非高并发服务；JSON 文件通常为 KB 到数 MB 级。
**关键路径延迟**: JSON 文件读取 + `JSON.parse` 是主要耗时；P99 目标为 < 100ms（文件 < 1MB 时）。
**扩展性瓶颈**: 文件大小无上限时，大 JSON 可能占用大量内存并阻塞事件循环。
**优化方案**: 限制单个文件大小（建议 10MB）；超出限制给出明确错误；不采用流式解析，因为 schema 校验需要完整对象。

#### ⚠️ 稳定性影响

**风险**: stdin 为空、文件不存在、TTY 误用、同时提供 stdin 和文件时冲突，均会导致调用失败或歧义。
**影响范围**: `plan`/`clarify`/`detail`/`replan`/`dev`/`test` 等大 JSON 输入命令。
**降级方案**: stdin 为空且为 TTY 时回退到 `--xxx-file`；文件不存在给出清晰错误；stdin 与文件同时存在时报冲突错误；非法 JSON 返回 exit ≠ 0。
**重试策略**: 无自动重试；调用方根据错误修正后重试。
**SLA 影响**: 无；CLI 单次调用，失败可重试。

#### ✅ 数据 / 并发 / 兼容性 / 可观测性

- 数据：JSON 透传给 engine，不修改数据契约。
- 并发：单次进程，无并发相关改动。
- 兼容性：stdin/文件双通道与 Unix 习惯一致，调用方可任选。
- 可观测性：输入方式不影响输出协议。

### Issue #7: 渐进式 action 退出语义 — 方案 A（严格分层）

#### ⚠️ 稳定性影响

**风险**: exit code 分层契约（exit 0 = 程序正常，exit ≥1 = 程序错误）与渐进式 gate fail 的语义若实现不清，调用方会混淆「需要重试的 gate fail」与「需要修复调用的 illegal_transition」。
**影响范围**: `dev`/`test`/`retrospect` 渐进式 action 及所有调用方 agent。
**缓解方案**: gate fail 时 exit 0 + `JSON.gatePassed=false`；非法状态转换/参数错误/内部异常时 exit ≥1；`protocol.ts` 统一映射错误类型到 exit code；文档明确 AC-2.4。
**残余风险**: 调用方未读 JSON 时可能误判；通过协议文档与测试约束。

#### ✅ 安全 / 数据 / 性能 / 并发 / 兼容性 / 可观测性

- 安全：exit code 语义不涉及权限。
- 数据：无数据格式变更。
- 性能：无额外开销。
- 并发：无并发相关改动。
- 兼容性：与 requirements C-1 一致。
- 可观测性：无日志/指标变化。

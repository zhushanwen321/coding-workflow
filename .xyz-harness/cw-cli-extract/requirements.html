① 澄清需求

coding-workflow engine 外置化为通用 CLI · ① 澄清需求 · business requirements

verdict: pass 来源: pi 扩展 @zhushanwen/pi-coding-workflow
核心结论
  • 将 coding-workflow 核心引擎从 pi 扩展中抽离为 agent-agnostic 独立 CLI（cw），运行时零 pi 依赖。
  • engine 核心逻辑层（状态机 / actions / checks / store / gates）源码原样复用，零行为变更；仅替换最外层 pi registerTool 薄壳为 CLI 适配层。
  • 任意 coding agent 仅靠「spawn 子进程 + 读 stdout JSON」即可走完完整 CW 流程（create→…→closeout）。
  • 行为等价通过「engine 单测原样 + CLI e2e 完整 lite 流程」验证；MCP / 多 runtime 形态留后续 topic。
  • 本轮范围 6 个用例：create / 推进流程 / 渐进式提交 / 查询状态 / 验证等价 / replan（append-only）。

业务目标

目标树

G1

engine 脱离 pi 独立运行

不引入 @mariozechner/pi-coding-agent / @earendil-works/pi-ai 任何运行时依赖，engine 全部 action 可经 CLI 触发并返回正确 ActionResult。

  • G1.1 核心逻辑层原样复用，零行为变更
  • G1.2 pi 耦合点被 CLI 适配层替换
G2

可被任意 agent 经子进程驱动

一个 coding agent 仅靠 spawn 子进程 + 读 stdout JSON 即可走完一条完整 CW 流程。

  • G2.1 稳定 action→命令映射 + 参数传递 + 结构化 JSON 输出
  • G2.2 现有 pi 扩展测试套件在 CLI 入口等价跑通
G3

engine 行为与 pi 扩展 100% 等价

同一组 action 调用序列，CLI 与 pi 扩展产生相同的状态流转 / gate 结果 / 持久化数据。

业务用例

UC-1: 启动 Topic

Actor: Coding Agent · 关联目标: G1

前置条件：agent 有可写的 workspacePath。后置状态：_cw.json 新增 topic 记录（status=created）；topicDir 目录就绪。

主流程：调 cw create --slug X --tier mid --objective "…" → 构造 ActionDeps → dispatch(handleCreate) → 输出含 topicId + nextAction 的 JSON。

  • AC-1.1 [正常]: 返回 JSON 含 topicId（cw-YYYY-MM-DD-slug）、status: "created"、nextAction 为 plan 或 clarify（依 tier）
  • AC-1.2 [异常]: slug 重复时 exit ≠ 0，stderr 含冲突信息，不产生半成品 topic
  • AC-1.3 [边界]: tier=lite → nextAction 指向 plan；tier=mid → 指向 clarify

UC-2: 推进流程（single-shot action）

Actor: Coding Agent · 关联目标: G1, G3

前置条件：topic 已存在且当前 status ∈ 该 action 的 expectedStatuses。后置状态：gate 通过则 status 流转；gateHistory 追加记录。

主流程：agent 产出阶段交付物到 topicDir → 调 cw plan --topicId X --plan-json … → 校验 JSON（typebox）→ dispatch → gate 跑机器检查 → 输出 JSON（exit 始终 0）。

  • AC-2.1 [正常]: gate 通过时 status 正确流转，nextAction 指向下一阶段
  • AC-2.2 [异常]: gate fail 时 status 不变，mustFix 列出 fail 项，exit 仍 0（结果在 JSON.gatePassed=false）
  • AC-2.3 [边界]: format 字段 ≠ tier 锁定值时被拒（D-003 tier 锁定），exit ≠ 0
  • AC-2.4 [协议]: exit 0 = 程序正常（gate pass/fail 都正常返回）；exit ≥1 = 程序错误（参数校验/illegal_transition/topic not found/内部异常）

UC-3: 提交进度（dev/test 渐进式）

Actor: Coding Agent · 关联目标: G2

前置条件：处于 developed/tested 进行态。后置状态：wave.committed / testCase.status 写入；达全 committed 则 gatePassed=true。

  • AC-3.1 [正常]: dev 提交后 nextAction.waves 反映已 committed 进度
  • AC-3.2 [异常-mid test]: 无效 commitHash（不可追溯已 committed 的 dev commit）→ 该 testCase 不标记 passed，gate fail
  • AC-3.3 [正常-lite test]: 用 judgeByExpected 机器重算（exact-match），丢弃 claimedStatus；缺 screenshotPath → failed
  • AC-3.4 [正常-mid test]: 信 claimedStatus + GitValidator 校验 commitHash 可追溯

test 的 lite/mid 是 engine 里分歧最大的 gate（strong-recompute vs medium-coverage），见架构 §5。

UC-4: 查询状态（CLI 新增便利命令，非 engine action）

Actor: Coding Agent / 开发者 · 关联目标: G2

dispatch 9 个 CwAction 无 status/list——CLI 层新增只读查询能力，非 G3 等价范围。nextAction 由 (tier,status) 反推。

  • AC-4.1 [正常]: status 输出 topic 的 status/gatePassed/waves/testCases 进度，与 engine loadTopic 一致
  • AC-4.2 [边界]: 不存在的 topicId → exit ≠ 0 + stderr 明确错误

UC-5: 验证行为等价

Actor: 开发者（CI / 手动）· 关联目标: G3

  • AC-5.1 [正常]: engine 单元测试在抽离后全绿（核心逻辑零改动应天然满足）
  • AC-5.2 [正常]: CLI e2e 至少覆盖一条完整 lite 流程（create→plan→dev→test→retrospect→closeout）
  • AC-5.3 [正常-replan]: CLI e2e 补 replan 路径（append-only 守卫生效），覆盖 9 个 action 中唯一回退语义
  • AC-5.4 [已知风险-mid]: mid 的 clarify.json/detail.json 经 CLI 入口无 e2e（D-004 scoping 后果）。缓解：detail 阶段补轻量 mid smoke 测试

UC-6: 追加计划（replan，append-only）

Actor: Coding Agent · 关联目标: G1, G3

前置条件：status ∈ {planned, developed}。replan 是 9 个 action 中唯一既非 single-shot 又非 dev/test 的 action，v1 仅 lite tier。

  • AC-6.1 [正常]: 追加新 wave/testCase 后，nextAction.waves 含新条目 + 旧 committed 条目不变
  • AC-6.2 [异常]: 破坏性变更（修改已 committed wave）→ 被拒，exit ≠ 0 + stderr 明确 append-only 语义
  • AC-6.3 [边界]: status=developed 时 replan 仍可（回退到 planned 追加），已 committed wave 跨 replan 不变

约束与不做

技术约束（仅记录不展开）

  • engine 源码原样复用，不得为 CLI 改核心逻辑（G3 等价性前提）
  • @sinclair/typebox 为唯一外部 schema 依赖（已是公共 npm 库，保留）
  • 运行时零 pi 依赖（@mariozechner/pi-coding-agent / @earendil-works/pi-ai 不得出现在 engine 或 CLI 运行时）
  • 纯 Node.js（无 Bun 特定 API 依赖，store 已用 node:fs）

协议约束

C-1

exit code 分层契约

exit 0 = 程序正常（gate pass/fail 都是正常返回，结果在 stdout JSON）；exit ≥1 = 程序错误（参数校验/illegal_transition/topic not found/内部异常，stderr 人类可读）。调用方 agent 读 JSON 判 gate 结果，不靠 exit code 区分 gate-fail。

C-2

存储路径完整结构

$CW_HOME/<encoded-cwd>/_cw.json（默认 ~/.cw/<encoded-cwd>/_cw.json），encodeCwd 规则原样继承 path-encoding.ts。per-cwd 隔离是 multi-workspace 正确性前提，不可扁平化。

C-3

worktree cwd 防护

CLI adapter 继承 pi 的 ADR-029 D1 防御：检测 process.cwd() 含 .cw-wt/ 时拒绝 fallback，强制显式 --workspace 或 CW_WORKSPACE_ROOT env。

不做（Out of Scope）

不做项原因 / 去向
MCP server 形态留后续 topic，本轮只做 CLI 子进程
多 runtime 适配本轮只验证一种 agent 接入，不做 adapter 抽象层
skills/workflows 平台无关化13 个 skill markdown 引用 cw action 名已 platform 无关；agent 侧加载/触发是 agent 问题
engine 核心逻辑修改任何行为变更需新开 topic
